Surface – Anonymous Feedback & Insights Tool (MVP Specification)

Overview
Surface is a web-based SaaS tool that helps small companies collect honest, mostly anonymous feedback from employees and turn it into clear, aggregated themes and actions for managers. The product focuses on enabling “venting” and structured peer/manager feedback while protecting anonymity and giving managers a simple dashboard view of team health.

The system uses:

Frontend: React + TypeScript

Backend: FastAPI (Python) exposing a REST API

Database: PostgreSQL

It is designed initially for a single-team pilot in a 20–100 person company, with a clear path to multi-team and full-organization support later.

Core goals

Make it easy and safe for employees to share honest feedback (rants + structured feedback).

Aggregate feedback into anonymized themes and simple metrics.

Give managers a clear, actionable dashboard (themes + example comments + actions).

Enforce anonymity via thresholds and de-identification so employees feel protected.

User roles

Employee

Logged in to submit one anonymous rant per cycle (optional tags) and structured feedback for every other team member (except themselves).

Sees: list of themes, manager’s actions per theme, and summarized anonymized feedback—as insightful as the system can make it, without revealing who said what.

Manager

Manages one team in the MVP.

Two separate views:

(1) View about themselves: similar to what employees see (themes, summarized feedback about the manager), with any extra details appropriate for the manager.

(2) Team overview: entire team plus structured feedback / ratings for each employee (aggregated, anonymized).

Logs “actions” per theme: multiple actions per theme allowed; managers can edit actions. Goal: commit to follow-through so employees see the loop close and feel heard.

Admin (HR or founder)

Has own login and admin UI. Sets up the organization and pilot team:

Imports users and teams systematically (e.g. CSV upload—standard for bulk import). Admin can choose team name when creating/importing.

Configures settings including anonymity thresholds (see Anonymity and thresholds below).

Views high-level metrics for the pilot team only (no identities). Pilot: one org, one team, one manager, rest employees.

High-level flows

Admin setup flow

Admin creates an organization and one pilot team.

Admin uploads a CSV with:

Employee ID

Name

Work email

Team

Manager ID (or a flag indicating this user is a manager)

The system stores these in PostgreSQL (users, teams).

Admin (or system default) sets:

The pilot team to activate.

The anonymity threshold (e.g., minimum 5 unique respondents).

A feedback cadence (e.g., every 2 weeks) for the pilot.

Feedback cycle flow (conceptual)
A “feedback cycle” is a time-bounded window (e.g., 2 weeks) for one team. Cycles repeat per team but do not have to be back-to-back (e.g. 2 weeks every 2 months).

Each cycle contains:

Anonymous rants (open text, optional tags). One rant per employee per cycle for MVP; later, option to edit submission before the cycle deadline.

Structured feedback: each employee gives structured feedback for every other person on the team (except themselves).

Cycle lifecycle:

Cycle opened → employees submit one rant and structured feedback for each teammate.

Cycle closes: automatically at end_date. Admin can close early or extend the window if needed.

Aggregation: runs automatically when the cycle closes. Admin or manager can also trigger it manually. Themes can be re-run if settings change (themes may then change).

Manager reviews dashboard, adds actions (multiple per theme; can edit). Employees see summary, themes, and manager’s actions.

Core backend behavior (FastAPI)

Auth & user management

Uses basic authentication (e.g., email + password with JWT).

Routes (examples):

POST /auth/register (for seeding or admin use in MVP)

POST /auth/login

GET /auth/me (returns current user)

Admin setup routes

POST /admin/users/import

Accepts uploaded CSV (or JSON for MVP) with user data. Creates users and teams in the database.

POST /admin/teams/{team_id}/cycles

Creates a new feedback cycle for a given team.

Employee feedback routes

POST /feedback/rant

Input: text + optional topic tags.

Backend: validates request (user authenticated, cycle open); calls AI service to de-identify text and infer theme/sentiment; stores a rant record linked to user and cycle. Raw text can be stored temporarily if needed but is short-lived; anonymized_text/theme/sentiment are the primary fields kept.

POST /feedback/structured (or batched)

Input: receiver_id, cycle_id, numeric scores (small informative set to start, e.g. support, communication; expand later), optional comments (what helped, what could improve).

Backend: validates that receiver is another team member (not self). Each employee gives structured feedback for every other person on the team. Stored with giver_id and receiver_id; giver identity never exposed in UI.

Aggregation & insights

POST /cycles/{cycle_id}/aggregate

Trigger: admin or manager can trigger manually. Also runs automatically when cycle closes.

Backend: queries all rants and structured_feedback for the cycle; themes are derived from both. Free-form themes (AI suggests labels). Groups by theme, sentiment summaries and counts; selects safe anonymized example comments per theme (respecting thresholds). Writes results into cycle_insights. Marks cycle status as “aggregated.” Re-running with different settings can change themes.

Manager dashboard routes

GET /cycles/{cycle_id}/themes

Returns: participation numbers; for each theme: theme, count, sentiment_summary, example_comments (if threshold met). If threshold not met for a theme: still show theme and count, with a note e.g. “Theme expressed but not enough responses to show anonymized example comments.”

GET /cycles/{cycle_id}/manager-summary

Returns: aggregated structured feedback about the manager (average scores, comment snippets), only when distinct givers ≥ manager-summary threshold (configurable separately from other thresholds).

POST /cycles/{cycle_id}/actions

Input: theme, action_text. Backend: stores manager’s action (multiple actions per theme allowed).

PUT or PATCH for existing action: managers can edit actions.

Employee view routes

GET /cycles/{cycle_id}/summary

Returns: top themes and their descriptions; the manager’s actions for each theme. Shows aggregated information only, no individual scores or comments tied to specific people.

AI integration (conceptual)

A dedicated service layer inside the backend (e.g., app/services/ai.py) encapsulates all AI calls.

Functions included:

deidentify_text(raw_text, employee_names) -> anonymized_text

classify_theme_and_sentiment(text) -> { theme, sentiment }

The MVP uses OpenAI or similar models to:

Remove names and obvious identifiers using a known list of employees.

Classify text into free-form themes (AI-suggested labels; not a fixed enum). Assign a coarse sentiment score. Themes are derived from both rants and structured feedback comments during aggregation.

For MVP, AI calls can happen synchronously in the request/response cycle. In later versions, heavy processing or full-cycle aggregation can move to background jobs.

Frontend behavior (React + TypeScript)

Pages / routes

Login page (/login)

Form to enter email and password.

On success, stores token (e.g., in memory/localStorage) and redirects user.

Redirect target depends on role:

Employee → /feedback

Manager → /dashboard

Employee feedback page (/feedback)

Rant section:

Text area for an anonymous rant.

Optional tags (workload, communication, leadership, tools, culture, onboarding, other).

Submit button calling POST /feedback/rant.

Structured feedback section:

Shows every other team member (all except self). For each: rating dimensions (e.g. support, communication on 1–5 scale), “What helped?” and “What could they do differently?” (or similar). Submits as batch or multiple calls to POST /feedback/structured.

Manager dashboard page (/dashboard)

Selector for current cycle.

Overview:

Participation summary.

Number of rants and structured feedback entries.

Themes section:

Cards or list of top themes (theme name, count, sentiment, trend if available).

Each theme expandable to show example anonymized comments.

“Add action” per theme (multiple allowed); POST /cycles/{cycle_id}/actions. Ability to edit existing actions.

Manager feedback section (separate view): summaries of how the manager is rated (structured feedback), when threshold satisfied.

Team summary (for employees)

After aggregation and manager actions, employees see: themes (with note when example comments withheld due to threshold), manager’s actions per theme, and summarized anonymized feedback—as insightful as possible.

Frontend integration with backend

A small API client wrapper handles: base URL for the API, attaching auth token in headers, and basic error handling.

Example API usage:

Login: POST /auth/login → store token.

Submit rant: POST /feedback/rant with { text, tags }.

Fetch themes: GET /cycles/{cycle_id}/themes.

Submit action: POST /cycles/{cycle_id}/actions.

Anonymity and thresholds

Multiple configurable thresholds (e.g. one for showing example comments per theme, one for manager summary, etc.). Each can be tweaked.

No per-person identities: manager and employees never see who gave individual feedback. Exact rant text is de-identified (AI + filters) before storage/display.

When a theme (or manager summary) is below the relevant threshold: still show the theme/count (or that feedback exists), with a note such as “Theme expressed but not enough responses to show anonymized example comments.”

Non-goals and constraints for MVP

No voice or audio input yet (text-only rants and comments).

No Slack/Teams or HRIS integrations.

No multi-team hierarchy UI beyond a single team + manager.

Minimal UI customization (focus on clean, usable defaults). Pilot: one org, one team, one manager, rest employees. Structured feedback: small fixed set of dimensions to start; expand later. Multiple configurable anonymity thresholds.

Product clarifications (alignment)

Cycles: Repeating per team; not necessarily back-to-back (e.g. 2 weeks every 2 months). One rant per employee per cycle for MVP; later, option to edit before deadline. Structured feedback: each employee gives feedback for every other team member (except themselves).

Closing and aggregation: Cycle auto-closes at end_date. Admin can close early or extend. Aggregation runs automatically on close; admin or manager can also trigger manually. Themes are free-form (AI-suggested), from both rants and structured feedback; re-running aggregation with different settings can change themes.

Manager view: (1) View about themselves (like employee view, with possible extra detail). (2) Team overview with ratings/feedback per employee (aggregated, anonymized). Actions: multiple per theme; managers can edit. Goal of actions: commit to follow-through so employees see the loop close.

Employee view: Themes, manager’s actions, summarized anonymized feedback—as insightful as possible. When a theme is below threshold: still show theme and count with a note (e.g. “Theme expressed but not enough to show anonymized example comments”).

Admin: Own login and UI; import users systematically (CSV is standard for bulk import). Admin can choose team name when creating/importing. Pilot: one org, one team, one manager, rest employees.

Thresholds: Multiple configurable thresholds (e.g. for example comments vs manager summary), each tweakable. Tags on rants: optional. Structured feedback dimensions: start with a small informative set (e.g. support, communication); add more later.

Intended development approach

Local development: backend and frontend run locally (possibly via docker-compose) and use a local Postgres instance.

Version control: all code is committed and pushed to GitHub in the “Surface” repo.

Deployment (later): backend is containerized and deployed to AWS (e.g., ECS Fargate); Postgres is hosted on AWS RDS; frontend is built and deployed via S3 + CloudFront or AWS Amplify.
